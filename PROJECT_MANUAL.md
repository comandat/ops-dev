# 📘 OpenSales - Project Manual & Agent Guidelines

> [!IMPORTANT]
> **To the AI Product Manager & Agent Team:** 
> Read this document completely before starting any development, planning, or QA tasks. This manual defines the system architecture, business logic, and strict workflows you must adhere to.

## 1. Executive Summary
**OpenSales** is a multi-tenant SaaS (Software as a Service) platform designed to centralize and automate e-commerce operations. It acts as an integration hub connecting merchants to multiple external marketplaces and billing systems (e.g., eMAG, Trendyol, FGO). 

The platform's primary goal is to synchronize product catalogs, automate dynamic pricing and stock updates, and aggregate orders from multiple sources into a single, unified dashboard for the merchant.

## 2. Technical Stack & Architecture
The project is structured as a **Monorepo** containing two distinct applications:

*   **Frontend (/frontend)**: Built with **Next.js** (Standalone mode), React, and Tailwind CSS. It communicates with the backend via REST APIs.
*   **Backend (/backend)**: Built with **NestJS**. It handles all business logic, scheduled cron jobs, and marketplace integrations.
*   **Database**: **PostgreSQL** managed via **Prisma ORM** (v6.19.2).
*   **Message Queue**: **Redis** with **BullMQ** (used for background tasks like syncing 10,000 products asynchronously).
*   **Authentication**: Managed by **Lucia Auth** (specifically `lucia-auth/adapter-prisma`).

### Infrastructure & Deployment (Railway)
The system is deployed on **Railway.app** using two separate environments:
1.  **Development Environment**: Listens strictly to pushes on the `dev` git branch. Isolated databases.
2.  **Production Environment**: Listens strictly to pushes on the `main` git branch. Live customer data.

## 3. Core Business Logic & Domains

### Multi-Tenancy (SaaS Model)
Everything in the database is scoped to a `Tenant`. A tenant represents a merchant company. All queries MUST filter by `tenantId` to prevent data leaks between clients.

### The Plugin Engine
Integrations with external platforms (eMAG, Trendyol) are built as dynamic "Plugins". The backend features a generic Plugin Engine that loops through active plugins for each tenant to pull orders or push product updates via Cron Jobs.
*   *Documentation for these APIs is located in the `/docs/plugin-integrations` folder.*

### Products & Offers
*   `Product`: The internal source of truth for an item (SKU, internal price, stock).
*   `ProductOffer`: The marketplace-specific representation of a product. A single product can have multiple offers (e.g., one for eMAG with a +10% price markup, one for Trendyol with different titles).

### Order Management
Orders are pulled from plugins into the `Order` and `OrderItem` models. Their lifecycle is tracked via an `OrderStatusHistory` table.

## 4. Workflows & Agent Responsibilities

> [!WARNING]
> We operate a strict Git Workflow. **NEVER push directly to `main`.** All development happens on the `dev` branch.

### 👑 Product Manager (PM) Agent
*   **Role**: Define the roadmap, analyze user requests, and break them down into actionable tasks for the Developers.
*   **Actionable**: 
    1. Read the user's prompt.
    2. Check the schema (`backend/prisma/schema.prisma`) to see if data models support the request.
    3. Delegate precise backend API tasks and frontend UI tasks to the Dev Agents.
    4. Ensure no features break the Multi-Tenant architecture.

### 💻 Developer Agents (Frontend & Backend)
*   **Role**: Write robust, clean code based on PM instructions.
*   **Actionable**:
    1. Always work on the `dev` branch.
    2. When adding database changes, modify `schema.prisma` and run `npx prisma db push` locally.
    3. Ensure all new Backend endpoints validate input (e.g., using `class-validator`) and restrict access to the user's `tenantId`.
    4. Frontend developers must maintain a clean, responsive UI (Tailwind) matching the SaaS aesthetic.

### 🧪 QA & Test Agents
*   **Role**: Prevent regressions and ensure application stability.
*   **Actionable**:
    1. The project includes a `qa-suite/` containing End-to-End (E2E) tests via Playwright.
    2. This suite is NOT deployed to production. It is run locally against the `dev` environment.
    3. Write new Playwright tests for every new feature generated by the Dev Agents. Run them before validating a feature.

### 🎨 UX/UI Agents
*   **Role**: Ensure the interface is intuitive, fast, and accessible.
*   **Actionable**:
    1. Focus on dashboard clarity. Merchants handle high volumes of data (tables with hundreds of orders/products).
    2. Design components that support loading states (spinners, skeletons) since API integration syncing takes time.

## 5. Getting Started for Agents
When receiving a development task:
1. Navigate to the branch: `git checkout dev`
2. Start the local stack using `docker compose -f docker-compose.saas.yml up`
3. Check the `backend/Dockerfile` and `frontend/Dockerfile` if changing infrastructure.
4. Document any new environment variables needed in `railway.toml`.
